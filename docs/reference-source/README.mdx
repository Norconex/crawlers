# Reference Docs Source

This folder contains the human-authored source for generated reference docs.

## Goal

Keep generated pages fully structured from schema while allowing contributors to edit narrative content and examples in MDX.

## Layout

- `<category>/<ComponentName>.mdx`
- Categories: `crawler`, `importer`, `committers`, `shared`

Examples:

- `importer/CountMatchesTransformer.mdx`
- `crawler/WebCrawlConfig.mdx`

## Frontmatter

Each schema-backed file should include frontmatter:

```yaml
---
component: CountMatchesTransformer
module: importer
propertyCoverage: all
---
```

Fields:

- `component`: Exact schema component name.
- `module`: Informational module/category label.
- `propertyCoverage`: `all` or `listed`.

## Missing Docs Highlights

To make missing content obvious during review, generated pages highlight gaps:

- Missing summary in property table renders as `[MISSING DOCS]`.
- TODO text in property table renders with a `[TODO]` prefix.
- Missing property details render as a `Missing Documentation` warning block.

Starter files also use warning callouts for missing summary/notes/property docs.

## Examples Authoring

The `## Examples` section is passed through as raw MDX into the generated 
reference page. You can use Markdown and the `<ConfigBlock>` component to 
present configuration in multiple languages (YAML, JSON, XML).

Each `<ConfigBlock>` is one standalone example. Title and description go in 
regular Markdown above the component — a `### heading` for a title, 
a paragraph for description.

### Providing code: three approaches

#### 1. Object only (recommended) — all three tabs auto-generated

Provide the config as a plain JavaScript object under `json`. YAML and XML are 
derived automatically.

```jsx
<ConfigBlock
  json={{ handler: { class: "MyHandler", someField: "value" } }}
/>
```

YAML and XML are serialized from the same object. Use this approach whenever 
the structure is the same across all three formats.

#### 2. Object with per-language overrides

When XML (or YAML) needs a different structure — for example, a wrapper 
element — provide it separately. Any language you supply explicitly overrides 
the auto-derived version.

```jsx
<ConfigBlock
  json={{ class: "MyCommitter", url: "http://host:9100" }}
  xml={{ committer: { class: "MyCommitter", url: "http://host:9100" } }}
/>
```

#### 3. Raw strings — full manual control

When the auto-generated output doesn't fit, provide strings directly. Template 
literals keep things readable without escaping.

```jsx
<ConfigBlock
  json={ `{ "handler": { "class": "MyHandler" } }` }
  xml={ `<handler><class>MyHandler</class></handler>` }
/>
```

You can mix approaches freely — for instance, use an object for `json` 
(auto-derives YAML) and a string for `xml`:

```jsx
<ConfigBlock
  json= { handler: { class: "MyHandler" } }
  xml={ `<handler><class>MyHandler</class></handler>` }
/>
```

### Multiple examples

Use one `<ConfigBlock>` per example, with Markdown headings and paragraphs for context:

```markdown
### Basic usage

Short description of what this example shows.

<ConfigBlock
  json={{ handler: { class: "MyHandler", field: "value" } }}
/>

### Advanced usage

Description of the more complex scenario.

<ConfigBlock
  json={{ handler: { class: "MyHandler", field: "value", extra: true } }}
  xml={{ handler: { class: "MyHandler", field: "value", extra: "true" } }}
/>
```

## Validation Rules

The generator validates docs-source files:

- Component must exist in schema.
- Property headings under `## Properties` should use backticks and match schema property names.
- `propertyCoverage: all` requires all schema properties to be present.
- `propertyCoverage: listed` requires `frontmatter.properties` and checks that entries exist in schema.

If validation fails, docs generation fails.

## Editing Workflow

1. Edit the matching file in this folder.
2. Commit the source changes in this repository.
3. The `crawlers-website` project rebuilds the generated reference pages and sidebar from this source during its site build.
